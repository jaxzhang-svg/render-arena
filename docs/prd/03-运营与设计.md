## 1. 运营需求

### 1.1 内容审核

#### 审核队列

- 标记潜在不当生成内容的系统
- 自动化审核 + 人工审核结合
- 优先级队列：高优先级（举报）> 中优先级（新用户）> 低优先级（老用户）

#### 社区准则

- **禁止内容**:
  - 违法内容（色情、暴力、仇恨言论等）
  - 侵犯版权的内容
  - 恶意代码（挖矿、钓鱼等）
  - 垃圾内容和滥用行为
- **允许内容**:
  - 创意的 Web 应用
  - 教育性内容
  - 技术实验和探索
  - 艺术性创作

#### 申诉流程

- 用户申诉审核决定的机制
- 申诉表单：原因、证据、联系方式
- 72小时内响应
- 申诉成功后恢复内容

### 1.2 基础设施准备

#### 可扩展性

- 处理峰值时段的并发生成请求
- 水平扩展：自动增加 Novita Sandbox 实例
- 负载均衡：多区域部署
- 缓存策略：CDN 缓存静态内容，Redis 缓存热点数据

#### 速率限制

- 在保持良好 UX 的同时防止滥用
- 用户级限制：每用户每小时最多 10 次生成
- IP 级限制：每 IP 每分钟最多 2 次生成
- 突发限制：短期突发流量控制

#### 成本监控

- 追踪每用户的 API 成本以确保可持续性
- 实时监控：LLM token 消耗、Sandbox 使用时长、S3 存储用量
- 成本预警：单用户成本超过阈值时预警
- 成本优化：缓存常见生成结果，减少重复调用

### 1.3 客户支持

#### 帮助文档

- 清晰的 FAQ 和使用指南
- 视频教程：快速入门（5 分钟）、高级功能（15 分钟）
- API 文档：开发者接口说明
- 最佳实践：提示词写作指南

#### 问题报告

- 便捷的错误或生成失败报告方式
- 自动错误收集：前端监控、后端日志
- 用户反馈表单：问题描述、复现步骤、环境信息
- 响应时间 SLA：24 小时内回复

#### 反馈循环

- 用户提出改进建议的机制
- 功能建议投票：用户可投票决定下一优先级功能
- 定期调研：月度用户满意度调查
- Beta 计划：早期用户优先体验新功能

## 2. 设计原则

### 2.1 代码 vs 结果

- **默认**: 不显示代码
- **重点**: 所有空间专用于视觉 Web App 结果
- **可选**: 代码视图可切换供感兴趣的用户查看

### 2.2 录制清洁度

- 确保录制捕获"干净画布"
- 从录制区域排除 UI 按钮和控件
- 设计录制友好的预览区域
- 所有非预览 UI 集中在顶部导航栏

### 2.3 移动端响应式

- **画廊**: 单栏卡片布局
- **Playground**: 对战模式使用 Tab 切换（非左右并排）
- **触控优化**: 控件大小适合手指操作（最小 44x44pt）

### 2.4 空状态

- **灵感卡片**: 未输入提示词时显示示例提示词
- **降低门槛**: 建议创意起点，让用户快速上手
- **示例内容**: "构建一个 3D 时钟"、"创建粒子系统"、"制作扫雷游戏"

### 2.5 非技术向用户

- **无技术术语**: 避免使用 "部署"、"构建"、"框架" 等技术词汇
- **视觉化展示**: 用截图、视频、动效代替文字说明
- **简单引导**: 每个功能都有清晰的操作指引
- **即时反馈**: 每个操作都有明确的视觉反馈
